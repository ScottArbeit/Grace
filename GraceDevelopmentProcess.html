Grace Development Process

A proposed Grace-native process derived from Brain2's disciplined agent workflow, adapted to Grace's current repository guidance, validation scripts, GitHub issue templates, and promotion-review world.

GitHub issue first Owned paths Vertical slices Fast validation Checkpoint commits Promotion-ready review

What Changes From Brain2

Brain2's process is strict GitHub issue, branch, PR, and merge-queue coordination. Grace should borrow the shape of that discipline while keeping Grace's local validation scripts, project guidance, and product language around promotion candidates, gates, attestations, and review reports.

Borrow From Brain2

  • Declare objective, context, owned paths, risk, validation, and done criteria before editing.
  • Keep work inside the declared write set unless the task record is updated.
  • Move in small vertical slices with focused tests and clear evidence.
  • Commit after each complete slice and keep PRs reviewable.
  • Treat docs drift, skipped validation, and residual risk as explicit review topics.

Keep Grace-Native

  • Use GitHub issues as the active task record for implementation work.
  • Use pwsh ./scripts/validate.ps1 -Fast as the normal validation gate.
  • Use -Full when Aspire, integration, emulators, or cross-service behavior changes.
  • Keep the fork and PR contribution path for outside contributors.
  • Map final integration language to Grace's promotion model: candidates, gates, attestations, and review reports.

Process Flow

Step-By-Step

Read the local contract

Start with repo-root AGENTS.md, then the closest project AGENTS.md. For most implementation work under src, read src/AGENTS.md and the project-specific guide such as src/Grace.CLI/AGENTS.md, src/Grace.Server/AGENTS.md, or src/Grace.Types/AGENTS.md.

Create or confirm the task record

Use the Grace agent task issue template when the user explicitly asks for an issue, asks to start tracked implementation, or otherwise requests tracker setup. When the user says Plan <work item>, plan the work in chat.

Turn the task record into a work contract

Before editing, write down objective, context, owned paths, forbidden or sensitive paths, risk surfaces, validation, docs impact, and definition of done. This is the Grace adaptation of Brain2's issue template.

Choose the workspace

For normal contribution work, use a focused branch and PR. For large or parallel work, prefer a task-owned worktree so unrelated work and existing dirty files stay isolated. If the user has asked to work in the current checkout, inspect git status first and avoid overwriting unrelated changes.

Classify the change

Pick a validation profile before coding: docs-only, pure domain, command/API boundary, actor/server behavior, SDK/client contract, UI/workflow, or deployment/runtime. Use the closest project guide to choose the test project and the repo-wide guidance to keep domain types in Grace.Types, helpers in Grace.Shared, and orchestration in the project-specific assembly.

Build one vertical slice at a time

Work through stable public boundaries: CLI command behavior, server endpoint behavior, Orleans actor behavior through server-surface tests, DTO/type contracts, SDK methods, or script behavior. Avoid broad horizontal rewrites unless the task is explicitly a refactor.

Use red, green, refactor evidence where behavior changes

Add or update a focused test that names the behavior, run it to see the meaningful failure, implement the smallest change, rerun the focused test, then refactor while green. Docs-only changes use Markdown, rendered artifact, or parser validation instead of a runtime RED step.

Run the fastest meaningful validation first

Start with the impacted test project, then promote to the repo validator when the change crosses project, fixture, build, or integration boundaries. The normal repo gates are: 

pwsh ./scripts/validate.ps1 -Fast
pwsh ./scripts/validate.ps1 -Full

Use -Full when Aspire integration, emulators, storage, Service Bus, Cosmos, Redis, deployment, or cross-service behavior matters.

Commit after each completed slice

Grace guidance already asks for granular git history. Commit each completed slice after its focused validation passes. The commit message should name the task and the behavior changed, not just the files touched.

Keep docs and agent guidance current

Update user-facing docs, README.md, CONTRIBUTING.md, or nearby AGENTS.md files when commands, behavior, APIs, dependencies, workflow, or project boundaries change. Keep guidance close to the project it affects.

Prepare review evidence

Before review, summarize changed paths, owned-path compliance, validation run, validation skipped with reason, docs impact, residual risks, and rollback or recovery notes. For outside contributions, include useful AI prompts from diagnosis or implementation in the PR description, as CONTRIBUTING.md requests.

Integrate by review, then promotion

In Git today, that means a focused PR. In Grace's own model, the equivalent destination is a promotion candidate evaluated by queue operations, gates, attestations, and review reports. The process should make a change easy to audit in either system.

Clean up after integration

After merge or promotion, verify the destination branch or reference contains the change, make sure no local work is stranded, remove task worktrees that are no longer needed, and update the task record with the final status and any follow-up.

Suggested Grace Task Contract

Use this as the lightweight replacement for Brain2's issue form when a Grace task does not already have a GitHub issue. It belongs in the issue body and should be carried into the pull request evidence. 

## Task Contract

Objective:
- One concrete behavior, docs, workflow, or infrastructure slice.

Context and evidence:
- Logs, files, symptoms, prior PRs, or design notes.

Owned paths:
- Files or directories this task may edit.

Forbidden or sensitive paths:
- Files or directories that require explicit expansion before editing.

Risk surfaces:
- Auth/secrets
- Storage/Cosmos/Service Bus/Redis/Aspire
- CLI public contract
- Server/API contract
- Orleans actor behavior
- SDK/client contract
- Docs/workflow
- No special risk expected

Validation:
- Focused command:
- Fast repo gate:
- Full/Aspire gate, if needed:
- Manual verification, if needed:

Definition of done:
- Behavior changed
- Tests or docs updated
- Fresh local review-only sibling subagent completed with no issues remaining; implementation subagents return a Ready
  For Review handoff instead of running nested codex review
- Review-only sibling subagents are allowed up to 10 minutes before stalled-progress analysis and must report back when
  complete so the parent/orchestrator can continue
- Validation recorded
- Standalone pull request comments use a well-templated Markdown Review/Fix format with clear headers, bold labels, a
  short high-level summary, detailed issue/fix text, and validation; do not put review-fix notes in the pull request
  body
- Review evidence prepared
- Follow-ups named

Validation Profiles

Docs Only

Markdown, HTML, guidance, or static documentation. Validate formatting, links or rendered output as appropriate. No code tests are needed unless docs describe generated behavior.

Domain Or Contract

Types, DTOs, validators, serializers, hashing, or shared helpers. Add focused tests in the matching Grace.Types.Tests or shared test surface.

CLI Or Server Boundary

Commands, endpoints, SDK calls, or actor-backed workflows. Prefer focused command/API/server-surface tests before broader validation.

Actor Behavior

Orleans behavior should usually be proven through server-surface integration tests instead of deep actor internals unless the project-specific guide says otherwise.

Deployment Runtime

Aspire, emulators, Azure settings, Docker, or scripts. Pair parser/script checks with validate.ps1 -Full or live evidence when runtime behavior is affected.

Review Or Promotion

Queue, candidate, gate, attestation, or report behavior. Capture deterministic evidence that a reviewer can replay from the CLI or API.
