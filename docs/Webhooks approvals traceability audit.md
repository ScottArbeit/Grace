# Webhooks and approvals traceability audit

<!-- markdownlint-disable MD013 MD060 -->

This audit closes the docs/OpenAPI/final-audit slice for epic
[#142](https://github.com/ScottArbeit/Grace/issues/142) and issue
[#152](https://github.com/ScottArbeit/Grace/issues/152).

## Status Classes

- **Implemented and proven**: implementation, tests, validation, and review evidence exist.
- **Implemented with incomplete proof**: implementation exists, but proof is narrower than the product criterion.
- **Intentionally deferred**: the product criterion is valid, but the current epic intentionally did not ship it.
- **Explicitly out of scope**: adjacent behavior that is not part of this epic.
- **Rejected**: a proposed shape was intentionally not implemented.

## OpenAPI and SDK Documentation Audit

The repo has a represented static OpenAPI surface under `src/OpenAPI`. Before this slice, that source did not include
the shipped webhook and approval routes. This slice adds static OpenAPI path/component files for:

- `/approval/policy/create|list|show|update|enable|disable|delete|evaluate`
- `/approval/request/list|show|approve|reject|history`
- `/webhook/rule/create|list|show|update|enable|disable|delete|test`
- `/webhook/delivery/list|show`

The implementation also has thin SDK modules in `src/Grace.SDK/Approval.SDK.fs` and `src/Grace.SDK/Webhook.SDK.fs`.
Those modules have inline XML comments and route strings, but the repo does not contain a generated SDK documentation
artifact for these APIs. No generated SDK documentation was updated because no generated SDK documentation surface is
represented in the repo.

There is no public `/approval/request/create` route, no public approval notification delivery query route, and no
approval notification delivery CLI command in the shipped surface. The internal `/approval/request/_seedGenerated`
workflow/test seam remains intentionally absent from the public OpenAPI additions in this audit.

## Validation and Review Evidence Summary

Every implementation PR in the epic recorded a successful `pwsh ./scripts/validate.ps1 -Fast` after its final fix pass.
Runtime webhook dispatch was additionally proven with focused `Grace.Server.Tests` webhook dispatch tests. The dispatch
slice did not run `validate -Full`; that is a residual integration risk because full Aspire/runtime validation is still
the stronger proof for hosted worker behavior.

## Residual Risks and Deferrals

- Approval notification delivery has contracts and notification URL validation, but no public notification-delivery
  query surface in this epic.
- Webhook dispatch has focused server-unit proof and Fast gate proof, but no recorded Full Aspire gate for the hosted
  runtime path.
- CLI JSON redaction is intentionally targeted to webhook rule and approval policy return shapes. Future DTOs that carry
  sensitive outbound URLs need explicit sanitizer coverage.
- `promotion-set list` remains a client-side composition of queue status and per-promotion-set fetches because no
  current `promotion-set/list` server endpoint exists.
