# Webhooks and approvals

Grace uses webhooks and approvals for two different jobs:

- A `webhook` is a post-event outbound HTTP notification rule. It observes committed Grace facts and never decides
  whether a workflow may continue.
- An `approval policy` is pre-transition gate configuration. It says that a matching workflow needs a responder before
  the transition can commit.
- An `approval request` is durable runtime decision state generated by Grace for one protected workflow attempt. It is
  not created directly by users.

The invariant is: webhooks observe committed facts, approval policies gate proposed transitions, and approval requests
are runtime decisions.

## Webhooks

A webhook rule belongs to a repository scope and can optionally target one branch. The rule stores:

- the event name and version, currently `promotion-set.applied` version `1`
- an inline destination URL
- a signing secret version
- retry policy settings
- rule status, such as `Enabled`, `Disabled`, or `Deleted`

Webhook delivery is server-side and post-event. When a committed event matches an enabled rule, Grace records a webhook
delivery with a stable dedupe key, stores the payload and rule snapshot, signs the outbound request, and attempts
delivery through the server worker path. Failed deliveries do not roll back or block the source workflow. Retry state is
recorded as delivery status such as `Pending`, `RetryScheduled`, `Failed`, or `DeadLettered`.

Grace sends these webhook headers on outbound delivery attempts:

- `x-grace-webhook-delivery-id`
- `x-grace-webhook-event-name`
- `x-grace-webhook-event-version`
- `x-grace-webhook-timestamp`
- `x-grace-webhook-signature-key-id`
- `x-grace-webhook-payload-sha256`
- `x-grace-webhook-signature`

Webhook management commands:

```powershell
grace webhook create --name applied --event promotion-set.applied --url https://example.com/grace/webhook
grace webhook list
grace webhook show --webhook <webhook-rule-id>
grace webhook update --webhook <webhook-rule-id> --name applied --event promotion-set.applied --url https://example.com/grace/webhook
grace webhook enable --webhook <webhook-rule-id>
grace webhook disable --webhook <webhook-rule-id>
grace webhook delete --webhook <webhook-rule-id>
grace webhook test --webhook <webhook-rule-id>
grace webhook deliveries --webhook <webhook-rule-id>
grace webhook delivery show --webhook-delivery <webhook-delivery-id>
```

bash / zsh:

```bash
grace webhook create --name applied --event promotion-set.applied --url https://example.com/grace/webhook
grace webhook list

```

The public command surface intentionally does not include `hook`, `subscription`, `notification rule`, or reusable
destination commands.

## Approval Policies

An approval policy defines a gate for a subject, currently used by `promotion-set.apply`. A policy stores:

- the repository and target branch scope
- the protected subject, such as a promotion set apply subject
- the required approval responder selector
- an optional notification URL
- timeout settings
- policy status, such as `Enabled`, `Disabled`, or `Deleted`

Approval policy commands:

```powershell
grace approval policy create --name release-approval --subject promotion-set.apply --required-responder user:alice
grace approval policy list
grace approval policy show --approval-policy <approval-policy-id>
grace approval policy update --approval-policy <approval-policy-id> --name release-approval --subject promotion-set.apply --required-responder user:alice
grace approval policy enable --approval-policy <approval-policy-id>
grace approval policy disable --approval-policy <approval-policy-id>
grace approval policy delete --approval-policy <approval-policy-id>
grace approval policy evaluate --subject promotion-set.apply
```

bash / zsh:

```bash
grace approval policy create --name release-approval --subject promotion-set.apply --required-responder user:alice
grace approval policy list
grace approval policy show --approval-policy <approval-policy-id>
grace approval policy update --approval-policy <approval-policy-id> --name release-approval --subject promotion-set.apply --required-responder user:alice
grace approval policy enable --approval-policy <approval-policy-id>
grace approval policy disable --approval-policy <approval-policy-id>
grace approval policy delete --approval-policy <approval-policy-id>
grace approval policy evaluate --subject promotion-set.apply
```

## Approval Requests

Approval requests are generated by Grace workflows. There is no public
`grace approval request create` command and no public `/approval/request/create` route.

For a promotion set apply gate, the request is scoped to the repository, target branch, promotion set, steps computation
attempt, approval policy id, and approval policy version. That scope prevents an approval for stale work from
authorizing a newer apply attempt.

An approval request can be `Pending`, `Approved`, `Rejected`, `Expired`, `Cancelled`, or `Superseded`. Approval and
rejection responses are idempotent by client decision ID and responder identity: the same decision can replay safely,
while conflicting decisions are rejected.

Approval request commands:

```powershell
grace approval request list
grace approval request show --approval-request <approval-request-id>
grace approval request approve --approval-request <approval-request-id> --reason "Looks good"
grace approval request reject --approval-request <approval-request-id> --reason "Needs fixes"
grace approval request wait --approval-request <approval-request-id>
grace approval request history --approval-request <approval-request-id>
```

bash / zsh:

```bash
grace approval request list
grace approval request show --approval-request <approval-request-id>
grace approval request approve --approval-request <approval-request-id> --reason "Looks good"
grace approval request reject --approval-request <approval-request-id> --reason "Needs fixes"
grace approval request wait --approval-request <approval-request-id>
grace approval request history --approval-request <approval-request-id>
```

The responder must have `ApprovalRequestRespond` on the stored request scope and must match the stored responder
selector. Body-supplied scope does not grant authority over a stored request.

## Promotion Set Approval Summary

Promotion sets do not gain a `PendingApproval` status. Instead, Grace derives a compact approval summary from the
current approval policy and approval request state. Promotion set show/list and apply output can include:

- approval state, such as `NotRequired`, `Pending`, `Approved`, `Rejected`, `Expired`, or `Stale`
- approval request id
- approval policy id
- required responder
- expiration when present
- next command guidance for pending requests

Use `grace promotion-set request-approval` to intentionally route through the same approval gate as
`grace promotion-set apply` without inventing a separate approval-request creation surface.

## Approval Notification Delivery

Approval policies can store an optional `NotificationUrl`. Grace validates that URL with the same outbound URL safety
policy used for webhook URLs. The shared contracts include `ApprovalNotificationDelivery` so approval notifications
remain distinct from webhook deliveries.

The current public repo surface does not expose approval-notification delivery query routes or CLI commands. Do not use
`webhook delivery` commands to inspect approval notification delivery state; those commands are only for webhook
deliveries.

## Unsafe Local-Development URLs

By default, outbound URLs must be absolute public HTTPS URLs. Grace rejects loopback, localhost, private, link-local,
metadata, multicast, unspecified, deprecated, transition, relay, and other special-purpose address ranges for public
targets.

Unsafe local-development URLs require all of these conditions:

- the server is running in the Development host environment
- the server opt-in setting `grace__webhooks__outbound_urls__allow_unsafe_local_development` is enabled
- the request or CLI command includes the explicit unsafe-local-development acknowledgement
- the target is loopback only

The CLI acknowledgement option is:

```powershell
--allow-unsafe-local-development
```

bash / zsh:

```bash
--allow-unsafe-local-development
```

This mode is only for local test callbacks such as loopback receivers. It should not be enabled in shared, staged, or
production environments. Even with unsafe local-development mode enabled, Grace still rejects private network,
link-local, metadata, and other non-loopback special-purpose targets.

## Security Notes

Webhook and approval notification URLs often contain secret material in paths or query strings. Grace redacts sensitive
URL material in logs and CLI JSON output for webhook rules and approval policies. Avoid putting long-lived credentials
in callback URLs; prefer short-lived receiver tokens or receiver-side signature verification.

Webhook deliveries are signed. Receivers should verify the signature, timestamp, payload hash, event name, event
version, and delivery id before trusting a payload. Webhook receivers should be idempotent because retries can happen
after transient failures.

Approval response authorization uses stored request state. A user cannot become the responder by changing request-body
scope, request id, or responder values.
