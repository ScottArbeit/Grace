Grace ContentBlock and Global Dedupe Design

Current design direction for Grace's content-addressable storage, large-file block aggregation, global dedupe hints, and block-level garbage collection.

This update supersedes the earlier chunk-centric reference-counting report that treated ContentChunk as a StoragePool-level actor or one-row-per-chunk entity.

Executive Summary

Grace should use ContentBlock as the StoragePool-wide storage, transfer, dedupe, and GC unit for large manifest-backed files. ContentChunk remains a logical chunk concept identified by a ChunkAddress, not a standalone persisted entity, actor, or per-chunk counter.

  • Small or regular files use repository-scoped WholeFileContent and do not enter global chunk/block dedupe.
  • Large files use FileManifest with ordered ContentBlockRange entries.
  • ContentBlockAddress is based on a compact block format version plus ordered ChunkAddress values.
  • ContentBlockMetadata is the mutable storage and lifecycle companion to a block.
  • ContentBlockMetadataRange existence means physical bytes are present; absence means the range has been reclaimed.
  • Global dedupe results are hints. ContentBlockMetadataRange is the authority for physical presence.
  • Grace v1 uses the first chunk plus modulo-256 key chunks for global dedupe discovery, returning protected candidate windows rather than raw chunk existence answers.
  • ManifestAddress includes per-range UncompressedSize; ContentBlockMetadata claims use whole-record optimistic concurrency; compaction starts at max(64 MiB, 10%) plus age and churn guards.

Xet-Inspired Shape

Grace is borrowing the Xet separation between logical chunks and larger block-like storage units. In Xet, chunks are grouped into xorbs, and file reconstruction refers to xorb chunk ranges. Grace keeps that shape, but uses its own names and adds stronger lifecycle metadata for compaction and hosted-storage cost control.

Xet concept Grace concept Design note
Chunk ContentChunk Logical piece of large file content, not a persisted actor or row.
Chunk hash ChunkAddress Canonical BLAKE3 hash string over uncompressed chunk bytes.
Xorb ContentBlock Ordered group of chunks used as the storage, transfer, and GC unit.
Xorb chunk range ContentBlockRange Manifest reconstruction range with UncompressedSize.
Shard CAS info Future dedupe index Batched candidate lookup, not authoritative storage presence.

The Xet lesson for Grace is not "never record per-chunk facts." It is "do not make each chunk a separately managed database object, service actor, or counter." Per-chunk facts may appear inside batched block metadata, manifest ranges, cache indexes, or dedupe shards when they earn their storage cost.

Content Model

Settled Vocabulary

Term Meaning Not this
ChunkAddress Exactly 64 hexadecimal characters representing BLAKE3 over uncompressed chunk bytes. Algorithm-prefixed multihash in the core CAS model.
ContentChunk A logical chunk of large file content named by ChunkAddress. A persisted entity, actor, row, or counter.
ContentBlock The ordered group of chunk addresses Grace stores, transfers, and compacts as a unit. A storage blob path or repository-owned object.
ContentBlockAddress BLAKE3 over the block address domain, compact block format version, and ordered chunk addresses. Physical layout, compression, per-chunk lengths, or repository policy.
ContentBlockRange One ordered reconstruction range inside a FileManifest. FileManifestTerm, xorb term, or raw block pointer.
ContentBlockMetadata Mutable storage and lifecycle description for a ContentBlock. Part of block identity or the block format specification.
ContentBlockMetadataRange A physically present ordinal range inside ContentBlockMetadata. A tombstone or per-chunk presence record.

Proposed Contract Shape

These records are design-level shapes, not implementation patches. They are intentionally compact and avoid one-object-per-chunk scaling.

Chunking and Thresholds

Grace should use Xet-sized chunks once content is on the manifest-backed path. The larger Grace-specific decision is whether a file enters that path at all, and that belongs to the repository's ManifestEligibilityPolicy.

Knob Settled direction Reason
ContentChunk target size 64 KiB Preserves fine-grained dedupe while ContentBlock handles storage aggregation.
ContentChunk minimum size 8 KiB Avoids tiny chunks and matches the Xet-sized range.
ContentChunk maximum size 128 KiB Bounds chunk size when the CDC boundary rule does not fire.
ContentBlock target About 64 MiB of uncompressed chunk content Keeps storage, transfer, and manifest ranges large while chunks remain small.
ManifestEligibilityPolicy Repository-level binary/text size policy Decides whole-file versus manifest-backed content without changing the fixed ChunkingSuite.
Binary default threshold Uncompressed size >= 1 MiB Binary files benefit from chunk/block dedupe earlier, even when they are not enormous.
Text default threshold Grace-defined compressed size >= 1 MiB Large source or generated text often compresses below 1 MiB and should remain WholeFileContent.

The binary/text split should reuse Grace's existing Git-style binary detector: scan the first 8 KiB of the file for a NUL byte, and classify the file as binary when one is present. Text compressed size must come from a Grace-defined eligibility compression profile, not from an incidental storage encoding.

Physical ContentBlock Format v1

The v1 block payload should be a compact framed format that can be validated independently from object-storage metadata while preserving cheap range reads and future compaction. 

magic/version
header length
chunk count
chunk table:
  chunk ordinal
  ChunkAddress
  uncompressed length
  encoded offset
  encoded length
  encoding flags
payload frames in ordinal order
block checksum/trailer

The chunk table is physical format metadata, not block identity. ContentBlockAddress still comes from the address-family context, compact block format version, and ordered ChunkAddress sequence. Encoded offsets, encoded lengths, compression choices, object keys, and ETags remain outside content identity.

Identity Boundaries

Included in ContentBlockAddress

  • Address-family domain/context.
  • Compact block format version.
  • Ordered ChunkAddress values.

Excluded from ContentBlockAddress

  • Per-chunk uncompressed length.
  • Physical offset and physical length.
  • Compression and encoding details.
  • Active manifest counts and lifecycle state.
  • Repository policy or manifest eligibility settings.

Grace can store a compact block format version with ContentBlock and ContentBlockMetadata. The full format specification lives in Grace itself and is not duplicated inside every block.

Metadata Range Semantics

ContentBlockMetadataRange should not store a presence enum or bool. Existence is the presence signal.

Condition Meaning Implication
Range exists and ActiveManifestCount > 0 Physical bytes are present and live. Compaction must preserve this range.
Range exists and ActiveManifestCount = 0 Physical bytes are present but reclaimable. Compaction may remove this range after a concurrency-safe recheck.
No range covers the ordinal span Physical bytes are not present. Dedupe reuse must treat the candidate as a miss.

TotalPhysicalBytes and ActivePhysicalBytes are stored because they are cheap and useful for compaction candidate selection. Reclaimable bytes are derived: 

ReclaimablePhysicalBytes = TotalPhysicalBytes - ActivePhysicalBytes

Global Dedupe Hints

A global dedupe hit is never enough to create a durable ContentBlockRange. Grace must confirm a matching ContentBlockMetadataRange exists and make that range active before a FileManifest can depend on it.

Settled v1 key-chunk rule

isKeyChunk(chunkAddress, chunkOrdinal):
  if chunkOrdinal == 0:
    return true

  value = littleEndianUInt64(last 8 bytes of BLAKE3 chunk digest)
  return value % 256 == 0

256 is the v1 SamplingRate. Grace should keep it as server policy surfaced in upload-session configuration, not as durable per-chunk state. The rule is deliberately denser than Xet's 1024 starting point because Grace's manifest path begins around 1 MiB, and early Grace workloads need useful discovery below 64 MiB.

  1. Client or server derives sparse key chunks from the local chunk sequence using the upload-session SamplingRate.
  2. Client or server asks the dedupe index for candidate block windows associated with those key ChunkAddress values.
  3. The dedupe index returns candidate ContentBlockAddress values, ordinal windows, and enough nearby sequence information to score contiguous runs.
  4. Grace prefers longer matching runs and rejects isolated chunk matches that would fragment reconstruction.
  5. Grace reads ContentBlockMetadata for the candidate block.
  6. If the metadata range is missing, the hint is stale and Grace treats it as a miss.
  7. If the metadata range is present, Grace increments or claims ActiveManifestCount for that range.
  8. Only after that claim succeeds may the new FileManifest record a ContentBlockRange.

The dedupe index can contain stale candidates. That is acceptable because it is a performance structure and locality hint, not the source of truth for physical presence.

Client Upload Flow

Grace clients can chunk, hash, locally dedupe, and propose manifest-backed content, but they do not get a free-standing ChunkAddress existence API. Grace Server remains authoritative for authorization, physical presence, range claims, and FileManifest finalization.

  1. Choose the content path. The client applies the Repository's ManifestEligibilityPolicy. Whole-file content stays repository-scoped. Only manifest-backed content enters this chunk, block, and global dedupe flow.
  2. Hash the logical file. The client reads the unencoded file bytes and computes the file-level FileContentHash. Storage compression, transfer compression, and object-storage layout do not change this identity.
  3. Chunk the unencoded bytes. The client applies the active ChunkingSuite and records the ordered local chunk sequence: file ordinal, byte offset, uncompressed length, and ChunkAddress.
  4. Hash each chunk. ChunkAddress is the lowercase BLAKE3 hash of the unencoded chunk bytes. The file path, repository policy, storage provider, and compression settings are not part of chunk identity.
  5. Deduplicate locally first. The client builds an in-memory map for repeated chunk runs within the file and current upload batch. This produces local block-building opportunities without any server lookup.
  6. Query sparse global anchors. The client selects the first chunk plus chunks whose BLAKE3 digest satisfies the modulo-256 key-chunk rule, then sends only those anchors in an authorized upload-session request. The sampling rate stays server-controlled upload-session policy, and the client does not query every ChunkAddress.
  7. Scan protected candidate windows. The server returns bounded candidate ContentBlock windows, not raw "exists" answers. Candidate chunk identities are protected with a short-lived response key, so the response does not expose raw neighboring chunk hashes. A conforming client matches by applying the same protection to the upload's local chunk sequence.
  8. Select contiguous reuse runs. The client scans the candidate windows for matching ordered runs, verifies chunk lengths, rejects stale or short matches, and prefers longer ranges that keep the future FileManifest compact.
  9. Ask Grace to claim reused ranges. Reuse runs remain proposals until Grace Server checks ContentBlockMetadata and claims the corresponding ContentBlockMetadataRange. Missing, compacted, unauthorized, expired, or over-limit candidates all collapse to "upload these bytes instead."
  10. Build and upload new ContentBlocks. The client packs all upload-required runs into ordered ContentBlock payloads of about 64 MiB of uncompressed chunk content, computes each ContentBlockAddress from the compact block format version and ordered ChunkAddress sequence, and uploads those block payloads before manifest finalization.
  11. Submit the FileManifest. The client submits a manifest proposal containing ManifestAddress, FileContentHash, total uncompressed size, ChunkingSuiteId, and ordered ContentBlockRange entries. The ranges must cover the file without gaps or overlaps.
  12. Finalize server-side. Grace Server validates that all referenced new blocks are uploaded, all reused ranges are still claimed, the ordered ranges reconstruct the expected chunk sequence, and the file-level hash and size match. Only then does the FileManifest become durable and eligible for a FileContentReference.

      BlockFormatVersion: int16

v1 response limits

  • MaxKeyChunksPerRequest = 256.
  • MaxCandidateWindowsPerKeyChunk = 4.
  • MaxWindowChunks = 256.
  • MaxResponseProtectedChunks = 16_384.
  • ResponseTtl = 5 minutes.
  • MinimumAcceptedReuseRun = max(8 chunks, about 1 MiB).

FileManifest invariants

  • Ordered ranges reconstruct the exact file byte stream.
  • No gaps, overlaps, or direct chunk references.
  • Every range points to a contiguous ordinal span inside one ContentBlock.
  • Each range records UncompressedSize for validation and accounting.
  • Manifest finalization is rejected if any referenced block range is not physically present and claim-active.

Existence-oracle guardrails

  • No exists(ChunkAddress) endpoint.
  • No per-chunk upload-skipping answer from global storage.
  • No raw chunk inventory returned in dedupe responses.
  • Empty or partial candidate results do not prove content absence.
  • Positive reuse is scoped to an authorized upload session and a successful range claim.
  • Reuse requires a server-side range claim and manifest validation; when in doubt, Grace uploads duplicate bytes.

This intentionally makes dedupe an optimization, not an authority. A client may save bandwidth when Grace can safely claim a range, but the correctness path is always available by uploading new ContentBlock payloads and finalizing a FileManifest from the resulting ranges.

Upload Session Lifecycle

The recommended v1 shape is one UploadSession per manifest-backed file, keyed by an UploadSessionId. Batch upload can create many sessions, but each session owns one FileContentHash, one expected total size, one ChunkingSuiteId, and one future FileManifest.

The existing actor patterns suggest this should be a GUID-keyed actor rather than a composite lookup record: Grace already uses GUID-keyed domain actors for durable workflow objects, composite string keys for alternate-name lookup actors, event lists for retry-safe state transitions, and reminders for delayed cleanup.

Started
  -> Discovering
  -> UploadingBlocks
  -> ClaimingRanges
  -> FinalizingManifest
  -> Finalized

Started/Discovering/UploadingBlocks/ClaimingRanges
  -> Abandoned
  -> Expired

Finalized/Abandoned/Expired
  -> RetentionPending
  -> StateDeleted

Durable session state

  • UploadSessionId and RepositoryId.
  • Authorized path or write scope used to create the session.
  • FileContentHash and expected total size.
  • ChunkingSuiteId and sampling policy snapshot.
  • Issued discovery response ids and expiration times.
  • New ContentBlockAddress upload intents.
  • Claimed reuse ranges and their claim expirations.
  • Final ManifestAddress, if finalized.

Idempotent commands

Upload-session commands need an explicit client operation id in addition to correlation id. Correlation ids are excellent diagnostics, but retries need a stable idempotency key that means "same client operation, replayed." 

type UploadSessionCommand =
    | Start of StartUploadSession
    | IssueDedupeDiscovery of DiscoveryOperationId * DedupeKeyChunk array
    | RegisterBlockUploadIntent of BlockUploadOperationId * ContentBlockAddress
    | ConfirmBlockUploaded of BlockUploadOperationId * ContentBlockAddress
            | ClaimReuseRanges of ClaimOperationId * ContentBlockRange array
            | FinalizeManifest of FinalizeOperationId * FileManifest
            | Abandon of AbandonOperationId
            | Expire of ExpireOperationId
            | DeletePhysicalState of DeletePhysicalStateOperationId

Retry and cleanup rules

  • Replaying the same operation id returns the same result if the original command already succeeded.
  • Expired discovery responses cannot be used to claim new ranges.
  • Expired or abandoned sessions release provisional range claims.
  • Uploaded-but-unfinalized blocks are cleaned up by delayed reminders unless another finalized manifest claims them.
  • Finalization is idempotent by ManifestAddress and rejects proposals that do not reconstruct the session's expected file hash and size.
  • Finalized, abandoned, and expired sessions schedule a repository-configured physical-state deletion reminder. The reminder clears only UploadSession actor state after the diagnostic and retry window.
  • The accepted FileManifest, ContentBlock payloads, ContentBlockMetadata, and contribution accounting are durable content state and are not deleted by upload-session cleanup.

Fragmentation Prevention

Xet's important warning is that chunk-level dedupe is useful only when it does not scatter one logical file across too many tiny block references. Grace should honor that by treating dedupe as a run selection problem, not as a per-chunk maximum-match problem.

Dedupe lookup shape

  • Query the first chunk plus modulo-256 key chunks instead of every ChunkAddress.
  • Return bounded, response-protected candidate metadata that describes nearby ContentBlock chunk sequences.
  • Let the uploader or server scan returned candidates for contiguous runs that match the file's local chunk sequence.
  • Treat key chunks as lookup anchors, not as special chunk identities or per-chunk lifecycle records.

Manifest construction rule

  • Prefer one ContentBlockRange over many short ranges when the content can be reconstructed either way.
  • Accept a dedupe run only when it clears a minimum run threshold, such as at least 8 chunks or roughly 1 MiB.
  • If an isolated chunk match would fragment the file, write that chunk into the next new ContentBlock instead.

This means Grace may intentionally duplicate a few chunks when that produces fewer, longer ContentBlockRange entries. The trade-off is deliberate: slightly less perfect dedupe can produce better reconstruction locality, lower metadata fan-out, and fewer read-time block fetches.

Presence-aware locality

A candidate run is usable only if the corresponding ContentBlockMetadataRange is still physically present and can be claimed active before manifest finalization. If a candidate block has been compacted and no longer covers the ordinal span, the run is stale and Grace treats it as unavailable.

Dedupe Index Maintenance

The dedupe index is a rebuildable discovery structure. Grace should write index entries only after block upload and manifest finalization succeed, and should remove or age out entries opportunistically when compaction reclaims ranges.

  • The index is populated from finalized FileManifest and ContentBlockMetadata facts, not from in-flight upload proposals.
  • Stale index entries are expected and safe; range claim checks against ContentBlockMetadata remain authoritative.
  • Compaction does not need to synchronously scrub every index entry. It can publish maintenance work or let future misses age stale candidates out.
  • Rebuilding the index from manifests and block metadata must be a valid recovery path.

Compaction and GC

Compaction rewrites physical payload placement without changing ContentBlockAddress or logical chunk ordinals. When Grace removes a reclaimable range, it removes the corresponding ContentBlockMetadataRange instead of storing a reclaimed tombstone.

Compaction selection

  • Rewrite only when ReclaimablePhysicalBytes >= max(64 MiB, 10% of TotalPhysicalBytes).
  • Treat hosted-storage transaction break-even as a floor, not the main selection rule.
  • Require reclaimable ranges to be at least 24 hours old before compaction can reclaim them.
  • Avoid blocks with active upload, finalization, range-claim, or compaction churn.
  • Prefer older or larger candidates after the hard floor.
  • Do not store permanent CompactionState on every block.

Concurrency rule

  • Reuse must claim active use before manifest finalization.
  • Compaction must recheck active counts before removing ranges.
  • Optimistic metadata concurrency is enough to make stale hints safe.
  • ContentBlockMetadata claims and compaction updates use a whole-record MetadataVersion or storage ETag.

Hosted storage cost floor

For Standard General Block Blob v2 in West US 2 using Hot GZRS, the current retail meters are approximately $0.0414 per GB-month, $0.004 per 10,000 read operations, and $0.1125 per 10,000 write operations. A simple one-read, one-write block rewrite costs about $0.00001165. That is offset by roughly 0.000281 GB-month, or about 295 KiB reclaimed for one month.

If the implementation charges work in 4 MiB transfer units, rewriting a 64 MiB physical block down to 48 MiB costs about $0.0001414 in read/write transactions. That is offset by roughly 3.5 MiB-month of Hot GZRS storage. Even then, raw transaction break-even is far below the operational threshold Grace should use for compaction.

Therefore, compaction should be gated by churn control, concurrency safety, candidate age, and batching, with transaction break-even used only as a sanity check.

Reference Counting Impact

The old per-ContentChunk counter design is retired. Chunk-level logical identity still matters for dedupe and verification, but lifecycle accounting should happen at the manifest and block-range level.

Old direction Current direction
ContentChunkActor with RepositoryReferenceCount. No per-chunk actor, row, or lifecycle counter.
CASTarget.ContentChunk chunkAddress. Large content flows through FileManifest, ContentBlock, and block ranges.
Manifest workflow fans out to distinct chunk counters. Manifest workflow adjusts active block ranges in bounded batches.
Chunk physical deletion reminder. Block compaction/removal based on ContentBlockMetadataRange liveness.

Implementation Boundary

Grace is not in production, so the CAS work does not need a compatibility migration path from the current path-shaped object upload model. The implementation can move directly to the right architecture: repository-scoped WholeFileContent for small or regular files, and FileManifest plus ContentBlock for eligible manifest-backed files.

Existing path-shaped upload behavior remains useful repo evidence, but it is not a migration constraint for the CAS design.

ADR Closure

These design questions are now captured in ADR 0001: Use ContentBlock-based content-addressed storage for large manifest-backed files.

  • UploadSession is a GUID-keyed actor per manifest-backed file. Batch upload is represented as many upload sessions.
  • ManifestContributionWorkflow is keyed by RepositoryId + ManifestAddress and owns repository contribution fan-out progress.
  • A save that introduces a manifest-backed FileVersion completes after the repository contribution increment intent is durably recorded. Full block-range fan-out can continue asynchronously with retry while blocking unsafe cleanup.
